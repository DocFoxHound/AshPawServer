PROJECT: 2D Multiplayer Animal Roleplay Game - SERVER REPOSITORY
PRIMARY GOAL: Build the authoritative multiplayer server for a cross-platform 2D overhead animal roleplay game.
LANGUAGE: C++
TARGET PLATFORMS: Linux first, with portability to macOS/Windows if needed for local dev/testing
REPOSITORY ROLE: This repo owns authoritative world simulation, player sessions, networking authority, state synchronization, interaction resolution, and persistence.

==================================================
1. SERVER MISSION
==================================================

This repository is the authoritative backend simulation for the game.

The server is responsible for:
- accepting and managing client connections
- authenticating or identifying sessions later
- owning the true world state
- simulating player and NPC state
- validating movement and interactions
- broadcasting relevant world updates
- routing chat and social systems
- saving/loading persistent state later

The server is NOT responsible for:
- rendering
- input handling
- audio
- client UI
- local visual effects
- camera
- sprite animation presentation

The server should be designed so that clients send requests and input, while the server decides what actually happens.

Core rule:
CLIENTS MAY REQUEST.
SERVER DECIDES.

==================================================
2. DESIGN PRINCIPLES
==================================================

Design the server with these principles:

1. Authoritative simulation
- the server owns true entity state
- position, interaction, and object changes are validated on the server

2. Deterministic-enough update flow
- clear update order
- fixed simulation tick
- avoid scattered gameplay mutation from random code paths

3. Simplicity first
- build a small, robust multiplayer core before advanced features

4. Network-conscious design
- packet types should be intentional
- do not send unnecessary data
- separate frequent updates from reliable event messages

5. Extensible but not overengineered
- avoid speculative abstractions
- structure code so new systems can be added without rewriting the core

6. Testability
- world simulation and validation logic should be testable without a graphical client

4. SERVER ARCHITECTURE OVERVIEW
==================================================

Core runtime model:

- one main server application
- one authoritative simulation loop
- one or more loaded maps or zones later
- connected clients mapped to sessions
- sessions mapped to player entities
- world state updated at fixed tick rate
- state deltas/snapshots broadcast to clients

High-level flow:

1. Server starts.
2. Network listener opens.
3. World/map loads.
4. Client connects.
5. Handshake occurs.
6. Session created.
7. Player entity created or restored.
8. Player sends input/requests.
9. Server validates and simulates.
10. Server broadcasts state/events.
11. On disconnect, session cleaned up and player saved later.

7. NETWORK PROTOCOL GUIDELINES
==================================================

The server should define and own the protocol contract.

Initial packet categories:

Connection:
- client hello
- server hello
- join accepted
- join rejected
- disconnect reason optional

Session/player lifecycle:
- player spawn
- player despawn
- identity update later

Gameplay input:
- movement input
- interaction request
- chat send
- emote/status request

Authoritative state:
- entity transform snapshot
- object state update
- interaction result
- chat broadcast

Guidelines:
- packets should be small and intentional
- use enums/opcodes clearly
- validate all incoming fields
- reject impossible or malformed input safely
- log protocol errors in development

9. SUGGESTED UPDATE ORDER PER TICK
==================================================

A recommended simulation tick order:

1. Poll/process inbound network messages
2. Apply validated input to session/player intent state
3. Run movement system
4. Resolve collisions
5. Run interaction system
6. Run NPC/object state updates if any
7. Build outbound replication data
8. Send snapshots/events
9. Perform cleanup for pending despawns/disconnects
10. Record metrics/logs as needed

Design note:
Avoid mutating world state from arbitrary packet handlers directly if possible.
Prefer packet handlers to stage intent/request data, then let simulation systems process it on tick.

12. SCOPE CONTROL RULES FOR SERVER DEVELOPMENT
==================================================

To keep the server project achievable:

- do not build for MMO scale
- do not build sharding/multi-zone infrastructure early
- do not build advanced account/auth before prototype needs it
- do not add complex permissions systems too soon
- do not optimize snapshot compression before basic replication works
- do not add multithreaded simulation unless single-threaded architecture becomes a proven bottleneck
- do not mix rendering or UI assumptions into server code

Priority order:
1. correctness
2. robustness
3. clarity
4. observability
5. performance optimization

13. CODING STANDARDS FOR THE SERVER AI AGENT
==================================================

Instructions for the coding agent working on the server repo:

- use modern C++ conservatively
- favor clear ownership and small focused classes
- keep networking, world simulation, and persistence separated
- avoid premature generic abstractions
- keep packet handling explicit and understandable
- make invalid states harder to represent where practical
- prefer simple data flow over clever architecture
- log important lifecycle and error events
- keep simulation deterministic enough for debugging
- build the smallest working server feature before extending it

When implementing a new feature:
1. identify which server phase it belongs to
2. identify what existing systems it depends on
3. implement minimal viable version
4. add validation and logging
5. explain how it will align with the client contract

When proposing refactors:
- explain what current limitation motivates the refactor
- preserve behavior unless a change is intentional
- avoid rewriting working systems for style alone

15. TRUE FIRST SUCCESS CONDITION
==================================================

The first real proof that the server is succeeding is not:
- “the architecture looks complete”
- “the protocol is future-proof”
- “the persistence layer exists”

The first real proof of success is:

“The server can accept two clients, create sessions for them, spawn them into the world, process movement input authoritatively, and broadcast stable state so both clients can see coherent multiplayer movement.”

After that, the next true proof is:

“The server can also route chat and resolve at least one interaction type authoritatively.”
