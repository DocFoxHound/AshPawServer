PROJECT: 2D Multiplayer Animal Roleplay Game
PRIMARY GOAL: Build a cross-platform overhead 2D multiplayer roleplay game in C++ while gaining experience in engine/framework design.
TARGET PLATFORMS: Linux, macOS, Windows
RECOMMENDED APPROACH: Build a custom C++ game framework on top of proven libraries instead of building a full engine entirely from scratch.

==================================================
1. HIGH-LEVEL PROJECT DECISION
==================================================

We are NOT building every low-level engine subsystem from nothing.

We ARE building:
- the gameplay architecture
- the world/entity system
- the rendering flow
- the networking model
- the client/server model
- the asset pipeline
- the game-specific systems

We SHOULD rely on libraries for:
- window creation
- input abstraction
- cross-platform timing
- graphics context creation
- image/audio loading
- networking transport
- build tooling

Recommended stack:
- Language: C++
- Build system: CMake
- Window/input/render bootstrap: SDL2 or SDL3
- Optional graphics path: SDL renderer first, OpenGL later if needed
- Networking: ENet
- Serialization: JSON initially, binary later if needed
- Debug UI: Dear ImGui
- Image loading: stb_image or SDL_image
- Audio: SDL_mixer or equivalent

Core design philosophy:
- keep scope tight
- prioritize playable milestones
- build a server-authoritative multiplayer architecture
- validate each layer before adding the next one
- do not overengineer early

==================================================
2. CORE PRODUCT VISION
==================================================

Game type:
- overhead 2D multiplayer roleplay game
- players control animal characters
- game emphasizes movement, interaction, chat, social presence, and world persistence over twitch combat

Initial playable concept:
- players connect to a shared map
- each player appears as an animal sprite
- players can move, chat, and interact with simple objects/NPCs
- server controls authoritative state
- client handles rendering and input

First true success state:
- two or more players can connect to a server
- move on the same map
- see each other in real time
- chat
- interact with at least one world object or NPC

PHASE 5: NETWORKING FOUNDATION
----------------------------------

Objective:
Introduce client/server architecture with the smallest possible multiplayer feature set.

Architecture decision:
Use a dedicated authoritative server.

Rules:
- clients send input and requests
- server owns true world state
- server broadcasts snapshots/events
- clients never decide authoritative positions or world outcomes

Deliverables:
- dedicated server executable
- client connection to server
- player join/leave flow
- player spawn synchronization
- movement synchronization
- basic connection recovery/disconnect handling

Tasks:
1. Integrate ENet into client and server.
2. Define packet protocol.
3. Define connection lifecycle:
   - connect
   - handshake
   - join accepted
   - spawn
   - disconnect
4. Create player session tracking on server.
5. Send input packets from client to server.
6. Simulate movement on server.
7. Send world snapshots or entity updates to clients.
8. Render remote players on client.
9. Handle disconnect cleanup.
10. Add debug network logs.

Success criteria:
- two clients can connect to one server
- each can see the other move
- movement is authoritative from server
- state stays coherent under normal use

Required protocol categories:
- connection/handshake
- spawn/despawn
- movement input
- transform snapshots
- chat
- interaction requests/results
- server notifications

6. TECHNICAL ARCHITECTURE GUIDELINES
==================================================

Game loop:
- use fixed timestep simulation
- keep rendering separate from simulation
- do not tie gameplay logic directly to render framerate

Suggested pattern:
- poll input
- accumulate time
- run simulation updates at fixed intervals
- render current state/interpolated state

Networking model:
- authoritative dedicated server
- clients send input or requests
- server simulates authoritative outcomes
- clients render received state plus smoothing

Why this model:
- easier anti-cheat baseline
- fewer desync issues
- cleaner authority rules
- more maintainable as features grow

Data boundaries:
- shared module contains packet/data definitions
- server should not depend on client rendering code
- client should not own authoritative gameplay state

Error-handling rules:
- fail loudly during development
- add useful logs for map loads, entity spawns, and network packets
- validate file/data inputs
- avoid silent failure

==================================================
7. MULTIPLAYER RULES OF THUMB
==================================================

Rule 1:
The client should send intent, not truth.

Examples:
GOOD:
- move up
- interact with object 42
- send chat message "hello"
BAD:
- my new position is X,Y
- door 5 is now open
- I own item 10 now

Rule 2:
The server decides:
- valid movement
- interactions
- object state
- player visibility rules
- persistence writes

Rule 3:
Use reliable delivery only where needed.
Reliable:
- chat
- spawns/despawns
- important interactions
- persistence-related messages
Unreliable or lower-priority:
- frequent movement/state snapshots

Rule 4:
Build packet types intentionally.
Do not send giant generic blobs unless necessary.

==================================================
10. SCOPE CONTROL RULES
==================================================

To keep this project achievable:

- finish one thin vertical slice before broadening scope
- never build a system just because “engines usually have one”
- avoid refactoring into complexity before there is real pressure
- do not build editor tooling before content creation pain is real
- do not build advanced combat if the game’s main value is social RP
- prefer boring, debuggable systems over clever systems

Every new feature should answer:
1. Does this improve the first playable multiplayer RP loop?
2. Can it wait until after two-player map/chat/interact works?
3. Will it increase maintenance complexity too early?

==================================================
11. CODING STANDARDS FOR THE AI AGENT
==================================================

Instructions for the coding agent:

- Use modern C++ but keep it straightforward.
- Favor clarity over abstraction.
- Avoid unnecessary template complexity.
- Avoid large inheritance hierarchies unless clearly justified.
- Separate client, server, and shared responsibilities cleanly.
- Use small focused classes/files.
- Add comments where design intent is not obvious.
- Prefer deterministic update logic where possible.
- Build minimal working implementations before generalized systems.
- Keep external dependencies limited and justified.
- Do not introduce premature optimization.

When implementing new systems:
- define the minimal version first
- make it compile and run
- validate behavior
- only then extend it

When proposing refactors:
- explain why the current structure is insufficient
- prefer incremental changes over rewriting whole subsystems

   - mention alternatives briefly
   - do not derail the project into unnecessary complexity
